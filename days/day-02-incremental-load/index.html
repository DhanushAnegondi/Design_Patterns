DEDP log / Chapter 2 / Day 02 · Incremental Loader

Stop reloading everything. Just chase the watermark.

When the source only ever grows and carries a column that says "this is new", you can ingest the delta instead of the whole thing. One stored high-water value does the bookkeeping. The catch: if that value is event time, late data slips through the cracks.

pattern: Incremental Loader delta column: event_time store: Amazon S3 local: moto mock

The shape of the problem

Same three blocks the book uses for every pattern.

Problem

Legacy producers still write visit events to a transactional database. The dataset grows constantly and every visit is immutable. Each run should ingest only the visits added since the last execution, not the whole table.

Solution

Pick a delta column (the visit's event_time) and remember the highest value ingested, the watermark. Each run takes rows where event_time > watermark, writes them, then advances the watermark.

Consequences

You move far less data, but now you hold state (the watermark) and you inherit a hazard: an event delivered late, behind the watermark, is skipped forever. That fix is a later pattern.

The mechanism, in code

The whole pattern is one filter and one forward-only update. From code/incremental_loader.py.

# ingest only rows beyond the stored high-water value
wm_before = self.get_watermark()
new_rows = [r for r in source_rows if r["event_time"] > wm_before]

if new_rows:
    self.s3.put_object(Bucket=bucket, Key=f"{prefix}/ingested_at={run_id}/data.csv", Body=to_csv(new_rows))
    wm_after = max(r["event_time"] for r in new_rows)
    if wm_after > wm_before:        # advance the watermark, never move it backward
        self._set_watermark(wm_after)
# a visit whose event_time is <= wm_before is never selected again: that is the late-data trap
